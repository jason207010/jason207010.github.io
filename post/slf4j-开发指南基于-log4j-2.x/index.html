使用maven构建

<dependency>
  <groupId>org.apache.logging.log4j</groupId>
  <artifactId>log4j-slf4j-impl</artifactId>
  <version>2.10.0</version>
</dependency>

此时会自动引入4个依赖jar包

包名
log4j-api-2.9.1.jar
log4j-core-2.9.1.jar
log4j-slf4j-impl-2.9.1.jar
slf4j-api-1.7.25.jar

配置

log4j 2.x默认使用classpath目录下log4j2.xml或log4j2.properties或log4j2.yaml或log4j2.json为配置文件，这里使用log4j2.xml作为配置文件

示例

<?xml version="1.0" encoding="UTF-8"?>
<!--每30秒检测一次配置文件是否改变，是则重新加载配置文件-->
<Configuration monitorInterval="30">
    <Properties>
    <Property name="logDir" value="logs"/>
    </Properties>
    <Appenders>
    <!--在控制台打印日志-->
    <Console name="stdout" target="SYSTEM_OUT">
        <PatternLayout pattern="%d{yyyy-MM-dd HH:mm:ss} [%p] %m%n"/>
    </Console>
    <!--每日打印日志-->
    <RollingRandomAccessFile name="dailyFile" fileName="${logDir}/current/dailyLog.log"
                 filePattern="${logDir}/$${date:yyyy-MM-dd}/dailyLog-%d{yyyy-MM-dd}_%i.log.gz">
        <PatternLayout pattern="%d{yyyy-MM-dd HH:mm:ss} [%p] %m%n"/>
        <Policies>
        <!--每天生成一个新的日志文件-->
        <TimeBasedTriggeringPolicy/>
        <!--在每天生成日志的基础上，如果日志大小超过100MB，则会对日志进行压缩存档，生成一个新的压缩文件-->
        <SizeBasedTriggeringPolicy size="100 MB"/>
        </Policies>
        <!--设置最大文件滚动数，超过最大值部分会被自动删除-->
        <DefaultRolloverStrategy max="999999"/>
    </RollingRandomAccessFile>
    <!--异步打印日志-->
    <Async name="asyncStdOut">
        <AppenderRef ref="stdout"/>
    </Async>
    <Async name="asyncDailyFile">
        <AppenderRef ref="dailyFile"/>
    </Async>
    </Appenders>
    <Loggers>
    <!--相加性设置为false，如果设置为true，则在打印日志的时候，还会调用一次RootLogger，这样就造成2次打印-->
    <Logger name="TestLogger" level="debug" additivity="false">
        <AppenderRef ref="stdout"/>
    </Logger>
    <Root level="debug">
        <AppenderRef ref="stdout"/>
        <AppenderRef ref="dailyFile"/>
    </Root>
    </Loggers>
</Configuration>

配置文件自动热更新

Logger

Logger级别

Level 描述
ALL 各级包括自定义级别
DEBUG 指定细粒度信息事件是最有用的应用程序调试
ERROR 错误事件可能仍然允许应用程序继续运行
FATAL 指定非常严重的错误事件，这可能导致应用程序中止
INFO 指定能够突出在粗粒度级别的应用程序运行情况的信息的消息
OFF 这是最高等级，为了关闭日志记录
TRACE 指定细粒度比DEBUG更低的信息事件
WARN 指定具有潜在危害的情况

对于标准级别它们关系如下：ALL < DEBUG < INFO < WARN < ERROR < FATAL < OFF。

RootLogger

在未特别指定Logger的情况下，log4j默认使用rootLogger作为Logger，因此一般都需要配置rootLogger

<Root level="debug">
    <AppenderRef ref="stdout"/>
    <AppenderRef ref="dailyFile"/>
</Root>

全异步Logger

全异步Logger依赖Disruptor框架，因此要在maven中引入Disruptor的依赖

Disruptor是一个开源的并发框架，，能够在无锁的情况下实现网络的Queue并发操作。

<dependency>
  <groupId>com.lmax</groupId>
  <artifactId>disruptor</artifactId>
  <version>3.3.7</version>
</dependency>

设置系统环境变量

System.setProperty("log4j2.contextSelector", "org.apache.logging.log4j.core.async.AsyncLoggerContextSelector");

无需修改log4j2.xml配置，此时的Logger就会自动变成全异步Logger

同步/异步混合Logger

同步/异步混合Logger同样依赖Disruptor框架，但无需设置”log4j2.contextSelector”系统环境变量，把log4j2.xml中标签改为，标签改为即可

Appender

AsyncAppender（异步Appender）

AsyncAppender并不处理日志，只是将日志缓冲到一个BlockingQueue里面去，并在内部创建一个工作线程从队列头部获取日志，之后将获取的日志循环记录到附加的其他appender上去，从而达到不阻塞主线程的效果。因此AsynAppender仅仅充当事件转发器，必须引用另一个appender来做事。

默认情况下，AsyncAppender使用java.util.concurrent.ArrayBlockingQueue，它不需要任何外部库。请注意，多线程应用程序在使用此应用程序时应小心谨慎：阻塞队列容易发生锁争用，我们的测试表明，当更多线程并发记录时，性能可能会变差。 考虑使用Async Loggers以获得最佳性能。

  • 配置参数

    参数名 类型 备注
    AppenderRef String 要异步调用的Appender的名称。 可以配置多个Appender
    blocking boolean true：appender将等待直到队列中有空闲位置。 false：在队列已满时将事件写入错误追加器。 默认值为true
    shutdownTimeout integer 在关闭时Appender应在等待队列中清除未完成的日志事件多少毫秒。 默认为零，表示永远等待
    bufferSize integer 队列的大小。 默认值为128。请注意，使用disruptor-style BlockingQueue时，此缓冲区大小必须为2的乘方
    errorRef String 由于追加器中的错误或由于队列已满而无法调用任何Appender，则调用这个Appender。 未指定时错误将被忽略
    filter Filter 筛选以确定事件是否应由此Appender处理。 通过使用CompositeFilter可以使用多个过滤器
    name String Appender的名称
    ignoreExceptions boolean 默认为true，当设置为false时，异常将被传播到调用者
    includeLocation boolean 打印location日志是非常耗费性能的（慢5-20倍）。默认不false，即不打印location信息
    BlockingQueueFactory BlockingQueueFactory 要使用的BlockingQueue的类型

  • 示例：

    <?xml version="1.0" encoding="UTF-8"?>
<Configuration monitorInterval="30">
    <Appenders>
    <Console name="stdout" target="SYSTEM_OUT">
        <PatternLayout pattern="%d{yyyy-MM-dd HH:mm:ss} [%p] %m%n"/>
    </Console>
    <Async name="asyncStdOut">
        <AppenderRef ref="stdout"/>
    </Async>
    </Appenders>
    <Loggers>
    <Root level="debug">
        <AppenderRef ref="asyncStdOut"/>
    </Root>
    </Loggers>
</Configuration>

    name String Appender的名称

FileAppender（文件Appender）

FileAppender是一个OutputStreamAppender，写入日志到fileName参数中指定的文件。 FileAppender使用FileManager（继承自OutputStreamManager）来实际执行文件I/O。虽然不能共享来自不同配置的FileAppender，但如果Manager可访问，则FileManager可以共享。例如，servlet容器中的两个Web应用程序可以有自己的配置，并且如果Log4j有一个共同的ClassLoader，则可以安全地写入同一个文件。

  • 参数配置

    参数名 类型 备注
    append boolean 默认为true：日志将写到文件的尾部，false：在写入日志之前清除文件
    bufferedIO boolean 默认为true：日志将被写到缓冲区，缓冲区已满或者设置了immediateFlush属性，则直接写入磁盘，不能和locking属性一起使用
    bufferSize int 当bufferedIO属性为true时，这个属性指的是缓冲区大小，默认为8192字节
    createOnDemand boolean Appender根据需要创建文件，默认为false
    filter Filter 筛选以确定事件是否应由此Appender处理。 通过使用CompositeFilter可以使用多个过滤器
    fileName String 日志文件名
    immediateFlush boolean 默认为true：每次写入后都进行flush确保数据写入磁盘，可能影响性能，即使设置为false，也会在一批日志写入后自动flush
    layout Layout 用来格式化LogEvent的布局。默认使用”％m％n”布局
    locking boolean true：只有持有文件锁的情况下才能进行I/O操作，这将影响性能，默认为false
    name String Appender的名称
    ignoreExceptions boolean 默认为true，当设置为false时，异常将被传播到调用者
    filePermissions String POSIX格式，在文件创建时设置文件属性。底层文件系统应支持POSIX文件属性视图。例如：rw——-或者rw-rw-rw-等等。
    fileOwner String 在文件创建时设置文件所有者，底层文件系统应支持文件所有者属性视图
    fileGroup String 在文件创建时设置文件组，底层文件系统应支持POSIX文件属性视图

  • 示例

    <?xml version="1.0" encoding="UTF-8"?>
<Configuration monitorInterval="30">
    <Properties>
    <Property name="logDir" value="logs"/>
    </Properties>
    <Appenders>
    <File name="file" fileName="${logDir}/current/file.log">
        <PatternLayout pattern="%d{yyyy-MM-dd HH:mm:ss} [%p] %m%n"/>
    </File>
    </Appenders>
    <Loggers>
    <Root level="debug">
        <AppenderRef ref="file"/>
    </Root>
    </Loggers>
</Configuration>

    RollingFileAppender（滚动文件Appender）

    RollingFileAppender是一个OutputStreamAppender，它写入fileName属性中指定的文件，并根据TriggeringPolicy和RolloverPolicy将文件滚动。 RollingFileAppender使用RollingFileManager（它继承自OutputStreamManager）来实际执行文件I/O并执行文件滚动。虽然来自不同配置的RolloverFileAppender不能共享，但如果Manager可以访问，RollingFileManagers可以共享。例如，一个servlet容器中的两个web应用程序可以拥有自己的配置，并且Log4j拥有同一个ClassLoader，则可以安全地写入相同的文件。

    RollingFileAppender需要一个TriggeringPolicy和一个RolloverStrategy。RolloverStrategy定义了文件如何滚动，而TriggeringPolicy决定了这些滚动操作是否应该被执行。如果没有配置RolloverStrategy，RollingFileAppender将使用DefaultRolloverStrategy。从log4j-2.5版本开始，可以在DefaultRolloverStrategy中配置自定义删除操作，以便在滚动时运行。从2.8版本开始，如果没有配置文件名，则将使用DirectWriteRolloverStrategy而不是DefaultRolloverStrategy。从log4j-2.9版本开始，可以在DefaultRolloverStrategy中配置一个自定义的POSIX文件属性视图操作，以便在滚动时运行，如果没有定义的话，将继承RollingFileAppender的POSIX文件属性视图。

    RollingFileAppender不支持文件锁定。

    • 参数配置
    参数名 类型 备注
    append boolean 默认为true：日志将写到文件的尾部，false：在写入日志之前清除文件
    bufferedIO boolean 默认为true：日志被写到缓冲区，缓冲区已满或设置了immediateFlush属性则直接写入磁盘，不能和locking属性一起使用
    bufferSize int 当bufferedIO属性为true时，这个属性指的是缓冲区大小，默认为8192字节
    createOnDemand boolean Appender根据需要创建文件，默认为false
    filter Filter 筛选以确定事件是否应由此Appender处理。 通过使用CompositeFilter可以使用多个过滤器
    fileName String 日志文件名
    filePattern String 归档日志文件名的模式。格式取决于RolloverPolicy。DefaultRolloverPolicy接受日期模式和代表整数计数器的％i
    immediateFlush boolean 默认为true：写入后进行flush确保数据写入磁盘，可能影响性能，即使设置为false，也会在一批日志写入后自动flush
    layout Layout 用来格式化LogEvent的布局。默认使用”％m％n”布局
    name String Appender的名称
    policy TriggeringPolicy 决定是否文件滚动的策略
    strategy RolloverStrategy 用于确定归档文件的名称和位置的策略
    ignoreExceptions boolean 默认为true，当设置为false时，异常将被传播到调用者
    filePermissions String POSIX格式，在文件创建时设置文件属性。底层文件系统应支持POSIX文件属性视图。例如rw-rw-rw-等等。
    fileOwner String 在文件创建时设置文件所有者，底层文件系统应支持文件所有者属性视图
    fileGroup String 在文件创建时设置文件组，底层文件系统应支持POSIX文件属性视图
    • 示例
    <?xml version="1.0" encoding="UTF-8"?>
<Configuration monitorInterval="30">
    <Properties>
    <Property name="logDir" value="logs"/>
    </Properties>
    <Appenders>
      <RollingFile name="RollingFileAppender" fileName="${logDir}/current/dailyLog.log"
           filePattern="${logDir}/$${date:yyyy-MM-dd}/dailyLog-%d{yyyy-MM-dd}_%i.log.gz">
        <PatternLayout pattern="%d{yyyy-MM-dd HH:mm:ss} [%p] %m%n"/>
        <Policies>
        <TimeBasedTriggeringPolicy/>
        <SizeBasedTriggeringPolicy size="100 MB"/>
        </Policies>
        <DefaultRolloverStrategy max="999999"/>
    </RollingFile>
    </Appenders>
    <Loggers>
    <Root level="debug">
        <AppenderRef ref="RollingFileAppender"/>
    </Root>
    </Loggers>
</Configuration>

RollingRandomAccessFileAppender

RollingRandomAccessFileAppender类似于标准的RollingFileAppender，除了它总是使用缓冲区（不能被关闭）并且在内部它使用ByteBuffer + RandomAccessFile而不是BufferedOutputStream。经过测试，比RollingFileAppender设置”bufferedIO = true”的性能提高了20-200％。RollingRandomAccessFileAppender将日志写入fileName参数中指定的文件，并根据TriggeringPolicy和RolloverPolicy将文件滚动。类似于RollingFileAppender，RollingRandomAccessFileAppender使用RollingRandomAccessFileManager 来实际执行文件I/O并执行滚动。虽然来自不同配置的RollingRandomAccessFileAppender不能共享，但如果Manager可以访问， RollingRandomAccessFileManagers可以共享。例如，一个servlet容器中的两个web应用程序可以拥有自己的配置，并且如果Log4j处于一个公共的ClassLoader中，则可以安全地写入相同的文件。

RollingRandomAccessFileAppender需要一个TriggeringPolicy和一个RolloverStrategy。RolloverStrategy定义了文件如何滚动，而TriggeringPolicy决定了这些滚动操作是否应该被执行。如果没有配置RolloverStrategy，RollingRandomAccessFileAppender将使用DefaultRolloverStrategy。从log4j-2.5版本开始，可以在DefaultRolloverStrategy中配置自定义删除操作，以便在滚动时运行。

RollingRandomAccessFileAppender不支持文件锁定。

  • 参数配置

    参数名 类型 备注
    append boolean 默认为true：日志将写到文件的尾部，false：在写入日志之前清除文件
    filter Filter 筛选以确定事件是否应由此Appender处理。 通过使用CompositeFilter可以使用多个过滤器
    fileName String 日志文件名
    filePattern String 归档日志文件名的模式。格式取决于RolloverPolicy。DefaultRolloverPolicy接受日期模式和代表整数计数器的％i
    immediateFlush boolean 默认为true：写入后进行flush确保数据写入磁盘，可能影响性能，即使设置为false，也会在一批日志写入后自动flush
    bufferSize int 缓冲区大小，默认为262144字节(256*1024)
    layout Layout 用来格式化LogEvent的布局。默认使用”％m％n”布局
    name String Appender的名称
    policy TriggeringPolicy 决定是否文件滚动的策略
    strategy RolloverStrategy 用于确定归档文件的名称和位置的策略
    ignoreExceptions boolean 默认为true，当设置为false时，异常将被传播到调用者
    filePermissions String POSIX格式，在文件创建时设置文件属性。底层文件系统应支持POSIX文件属性视图。例如rw-rw-rw-等等。
    fileOwner String 在文件创建时设置文件所有者，底层文件系统应支持文件所有者属性视图
    fileGroup String 在文件创建时设置文件组，底层文件系统应支持POSIX文件属性视图

  • 示例

    <?xml version="1.0" encoding="UTF-8"?>
<Configuration monitorInterval="30">
    <Properties>
    <Property name="logDir" value="logs"/>
    </Properties>
    <Appenders>
      <RollingRandomAccessFile name="dailyFile" fileName="${logDir}/current/dailyLog.log"
                   filePattern="${logDir}/$${date:yyyy-MM-dd}/dailyLog-%d{yyyy-MM-dd}_%i.log.gz">
        <PatternLayout pattern="%d{yyyy-MM-dd HH:mm:ss} [%p] %m%n"/>
        <Policies>
        <TimeBasedTriggeringPolicy/>
        <SizeBasedTriggeringPolicy size="100 MB"/>
        </Policies>
        <DefaultRolloverStrategy max="999999"/>
    </RollingRandomAccessFile>
    </Appenders>
    <Loggers>
    <Root level="debug">
        <AppenderRef ref="dailyFile"/>
    </Root>
    </Loggers>
</Configuration>

    Filter

    ThresholdFilter（日志等级过滤器）

    如果LogEvent中的级别与配置的级别相同或更高，则此过滤器将返回onMatch结果，否则返回onMismatch值。例如，如果ThresholdFilter配置了Level ERROR，并且LogEvent包含Level DEBUG，则会返回onMismatch值，因为ERROR事件比DEBUG更高级。

    • 参数配置
    参数名 类型 备注
    level String 要匹配的级别
    onMatch String 过滤器匹配时的行为，可以是ACCEPT、DENY或者NEUTRAL，默认值为NEUTRAL
    onMismatch String 过滤器不匹配时的行为，可以是ACCEPT、DENY或者NEUTRAL，默认值为NEUTRAL
    • 示例

    虽然rootLogger配置的等级是DEBUG，但ThresholdFilter过滤器定义了只有当INFO时才会ACCEPT打印日志。因此只会打印出INFO等级以上的日志。

    <?xml version="1.0" encoding="UTF-8"?>
<Configuration monitorInterval="30">
    <Appenders>
    <Console name="stdout" target="SYSTEM_OUT">
        <PatternLayout pattern="%d{yyyy-MM-dd HH:mm:ss} [%p] %m%n"/>
        <ThresholdFilter level="INFO" onMatch="ACCEPT" onMismatch="DENY"/>
    </Console>
    </Appenders>
    <Loggers>
    <Root level="DEBUG">
        <AppenderRef ref="stdout"/>
    </Root>
    </Loggers>
</Configuration>

CompositeFilter（复合过滤器）

CompositeFilter提供了一种指定多个过滤器的方法。它作为一个过滤器元素添加到配置中，并包含其他过滤器。 filters元素不接受任何参数。

  • 示例

    只有日志级别在INFO以上，并且在05:00:00~05:30:00时间段的日志才会被打印出来

    <?xml version="1.0" encoding="UTF-8"?>
<Configuration monitorInterval="30">
    <Filters>
    <ThresholdFilter level="INFO" onMatch="NEUTRAL" onMismatch="DENY"/>
    <TimeFilter start="05:00:00" end="05:30:00" onMatch="ACCEPT" onMismatch="DENY"/>
    </Filters>
    <Appenders>
    <Console name="stdout" target="SYSTEM_OUT">
        <PatternLayout pattern="%d{yyyy-MM-dd HH:mm:ss} [%p] %m%n"/>
    </Console>
    </Appenders>
    <Loggers>
    <Root level="DEBUG">
        <AppenderRef ref="stdout"/>
    </Root>
    </Loggers>
</Configuration>

    Layout

    PatternLayout

    如果想生成基于模式的特定格式的日志信息，那么可以使用PatternLayout格式化日志信息

    • 模式转换字符
    转换字符 表示的意思
    c 用于输出的记录事件的类别。例如，对于类别名称”a.b.c” 模式 %c{2} 会输出 “b.c”
    C 用于输出呼叫者发出日志请求的完全限定类名。例如，对于类名 “org.apache.xyz.SomeClass”, 模式 %C{1} 会输出 “SomeClass”.
    d 用于输出的记录事件的日期。例如， %d{HH:mm:ss,SSS} 或 %d{dd MMM yyyy HH:mm:ss,SSS}.
    F 用于输出被发出日志记录请求，其中的文件名
    l 用于将产生的日志事件调用者输出位置信息
    L 用于输出从被发出日志记录请求的行号
    m 用于输出使用日志事件相关联的应用程序提供的消息
    M 用于输出发出日志请求所在的方法名称
    n 输出平台相关的行分隔符或文字
    p 用于输出的记录事件的优先级
    r 用于输出毫秒从布局的结构经过直到创建日志记录事件的数目
    t 用于输出生成的日志记录事件的线程的名称
    x 用于与产生该日志事件的线程相关联输出的NDC（嵌套诊断上下文）
    X 在X转换字符后面是键为的MDC。例如 X{clientIP} 将打印存储在MDC对键clientIP的信息
    % 文字百分号 %%将打印％标志
    • 示例
    <Console name="stdout" target="SYSTEM_OUT">
    <PatternLayout pattern="%c %C %F %l %L %M %T %t %x %highlight{%d{yyyy-MM-dd HH:mm:ss} [%p] %m%n}"/>
</Console>

其他Layout

除了以上Layout，还有HTMLLayout、JSONLayout、XMLLayout等，由于不常用，不再赘述。

使用

package com.test;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
public class MyTest {
    private static final Logger LOGGER = LoggerFactory.getLogger(MyTest.class);
    public static void main(String[] args) {
    System.setProperty("log4j2.contextSelector", "org.apache.logging.log4j.core.async.AsyncLoggerContextSelector");
    LOGGER.trace("{}", "hello world!");
    LOGGER.debug("{}", "hello world!");
    LOGGER.info("{}", "hello world!");
    LOGGER.warn("{}", "hello world!");
    LOGGER.error("{}", "hello world!");
    }
}
